* PLAIN TALK

大家都想用大语言模型做各种文字任务，但模型越大越贵，尤其上线后每次回答都要烧算力。本文训练了一组 70 亿到 650 亿参数的模型，只用公开数据喂到 1.0T 到 1.4T token（token 就是把文本切成的小碎片），证明把较小模型训练得更久，往往能以更低的推理成本达到甚至超过更大模型的效果。结果上，LLaMA-13B 在多数基准上超过 GPT-3 175B，LLaMA-65B 接近 Chinchilla-70B，并在部分任务上超过 PaLM-540B。

* NAPKIN FORMULA

#+begin_example
+-----------------------------------------------------------+
| TotalCost = TrainCost + ServeCost                         |
| TrainCost = a * Params * Tokens                           |
| ServeCost = b * Params * Queries                          |
| same quality  choose smaller Params and larger Tokens     |
+-----------------------------------------------------------+
#+end_example

上式里 Params 是参数量（模型大小），Tokens 是训练时读过的 token 数，Queries 是线上被调用的次数，a 和 b 反映硬件与实现效率；当线上调用次数很大时，ServeCost 往往主导，因此更小的 Params 常比更大的 Params 更实惠。

* PROBLEM

*痛点定义*: 在给定线上推理预算（显存 + 延迟 + 算力）下，如何拿到尽可能强的通用语言模型。

*前人困境*:
- 只盯着把模型做大  容易带来上线成本爆炸
- 顶级模型训练细节和数据不透明  外界复现和比较困难
- 训练算力固定时  该把钱花在更多参数还是更多数据  长期缺少可操作的配方

* INSIGHT

*核心直觉*: 把服务成本当成硬约束  与其训练一个巨无霸  不如训练一组更小的模型并把它们训练到更久  让更多场景能用得起。

*关键步骤*:
1. *训练配方*  7B 到 65B 统一架构  小模型 1.0T tokens  大模型 1.4T tokens  让模型把公开语料吃得更充分
2. *结构小改动*  RMSNorm（更简洁的归一化）+ SwiGLU（更强的前馈门控）+ RoPE（旋转位置编码  让模型知道顺序）  在不改变 Transformer 主体的前提下提升性价比
3. *工程加速*  高效注意力实现 + 激活检查点 + 并行训练  把训练吞吐推到足够高  让长时间训练可承受

* DELTA

*vs SOTA*:
- *推理性价比*  LLaMA-13B 在多数零样本（不给示例直接答）基准上超过 GPT-3 175B  用更小模型达到更强效果
- *强度上限*  LLaMA-65B 与 Chinchilla-70B 接近  并在部分任务上超过 PaLM-540B
- *可复用性*  只用公开数据训练  并向研究界发布模型权重  降低重复训练成本
- *可调教性*  简单指令微调版 LLaMA-I 在 MMLU（多学科选择题测评）5-shot（给 5 个示例）上达到 68.9  展示了不错的可对齐潜力

术语小抄  zero-shot 不给示例直接答  few-shot 给少量示例再答

*新拼图*:
- 把一个很现实的目标说清楚并做成了配方  不是只追求最大分数  而是追求在真实部署约束下的最优点
- 给了一个可被复现的强基座  让研究可以把算力更多花在对齐与应用  而不是反复从零预训练

* CRITIQUE

*隐形假设*:
- *公开数据足够好*  假设经过过滤与去重后  公开语料的质量和覆盖能支撑强通用能力  但数据污染与版权边界很难彻底证明
- *评测能代表真实使用*  假设零样本与 few-shot 基准能近似线上体验  但长对话  工具使用  多轮纠错等能力未被系统衡量
- *推理预算主要由参数量决定*  实际还受上下文长度  注意力缓存（生成时存历史信息）  batch  并发等影响  小模型并不总是更省

*未解之谜*:
- *安全与毒性*  规模变大时有更强生成能力也可能更容易输出有害内容  论文只做了初步测量与讨论  缺少可靠的工程化闭环
- *真正的开放*  虽然发布权重给研究界  但门槛与许可限制仍影响社区复现与下游扩散
